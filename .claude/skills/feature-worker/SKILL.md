# Feature Worker

指定されたIssueを1件実装する。`AGENTS.md` に定義されたTDD開発フローを厳守すること。

## 呼び出し方

```
/feature-worker #44
/feature-worker #44 メッセージピン留め機能の実装
```

Issue番号を省略した場合は、現在のコンテキスト（会話履歴・ブランチ名等）から判断する。

---

## 実行フロー

### Step 0: 事前調査

以下を順に実行する。

```bash
gh issue view {番号}        # Issue詳細を取得
```

- `AGENTS.md` を読む（開発規約の確認）
- `CLAUDE.md` を読む（プロジェクトルールの確認）
- `db/schema.hcl` を読む（DBスキーマの確認）
- Issue要件に関係するコンポーネント・APIルート・テストファイルを特定して読む
- `doc/` 配下の関連仕様書があれば読む（特に `doc/snackbar-spec.md`）
- バックエンドのテストを実装する場合は `doc/db-test-guide.md` を読む

**実装前に影響ファイルをすべて把握すること。読まずに実装しない。**

---

### Step 1: ブランチ作成

```bash
git checkout -b feature/{機能名}/#{issue番号}
```

命名例:
- `feature/pin-message/#44`
- `feature/bookmark/#45`
- `feature/mention-badge/#46`
- `feature/direct-message/#43`

mainブランチから分岐していることを確認する。

---

### Step 2: テスト項目作成（構造のみ・中身なし）

**バックエンド** (`packages/server/src/__tests__/`)、**フロントエンド** (`packages/client/src/__tests__/`) にテストファイルを作成する。

ルール:
- `describe` と `it` のネスト構造のみを記載する
- アサーションは **一切書かない**。中身は `// TODO` のみ
- テストの説明文は日本語で書く
- ファイル冒頭にテスト対象・戦略のコメントを付与する

```ts
// テスト対象: ピン留めサービス
// 戦略: ピン留め操作のビジネスロジックとDB操作を検証する
describe('ピン留め機能', () => {
  describe('ピン留め追加', () => {
    it('メッセージをピン留めできる', () => {
      // TODO
    });
    it('同じメッセージを二重にピン留めするとエラーになる', () => {
      // TODO
    });
  });
});
```

テストファイルを保存した後、以下のどちらかに従う。

**単独実行時（`/feature-worker` を直接呼び出した場合）:**
ここで停止してユーザーに確認を求める。

```
テスト項目を作成しました。確認してください:
- packages/server/src/__tests__/xxx.test.ts
- packages/client/src/__tests__/xxx.test.tsx

問題なければ実装を進めます。修正があればお知らせください。
```

**並列実行時（オーケストレーター経由でサブエージェントとして動作中）:**
ユーザーへの確認は行わない。テストファイルのパスと内容をオーケストレーターに返答して停止する。ユーザー確認はオーケストレーターが全Issue分まとめて行う。

```
## テスト項目作成完了（Issue #{番号}）

### 作成ファイル
- packages/server/src/__tests__/xxx.test.ts
- packages/client/src/__tests__/xxx.test.tsx

### 内容
（各ファイルの全文を貼り付ける）
```

---

### Step 3: テストコード実装（ユーザー確認後）

ユーザーの承認を得てから、Step 2 で定義した項目のテストロジックを実装する。

テスト設計方針は AGENTS.md「テスト設計方針」セクションに従う。バックエンドのDBテストパターンは `doc/db-test-guide.md` を参照。

---

### Step 4: プログラム実装

以下の順で実装する:

1. **DBスキーマ変更が必要な場合**: `db/schema.hcl` を編集し `atlas schema apply --env local --dry-run` で差分確認後に `atlas schema apply --env local` で適用する
2. **共有型定義**: `packages/shared/src/types/` に型を追加する（必要な場合）
3. **バックエンド**: APIルート → サービス層 → Socket.IO ハンドラの順で実装する
4. **フロントエンド**: コンポーネントを実装する（React 19 の `use()` + `<Suspense>` パターンに従う）

**React 19 実装ルールは CLAUDE.md「フロントエンド開発ルール」に従う。**

**実装中の禁止事項:**
- テストを変更して通すことは原則禁止
- 既存テストのアサーションを弱める・削除することはユーザー承認なしに禁止

---

### Step 5: テスト実行・確認

```bash
npm run build   # 型チェック・ビルド確認
npm run test    # テスト全通過確認
```

どちらかが失敗した場合は修正して再実行する。**両方が成功するまでPRを作成しない。**

既存テストを変更した場合は理由を記録しておく（完了報告に含める）。

---

### Step 6: コミット＆PR作成・更新

コミット規約（プレフィックス + 日本語）:
```bash
git add {変更ファイル}
git commit -m "feat: {機能の説明}"
git push -u origin {ブランチ名}
```

#### 新規PRを作成する場合

PR作成は `.github/PULL_REQUEST_TEMPLATE.md` のテンプレートに従う:

```bash
gh pr create --title "{タイトル}" --body "$(cat <<'EOF'
## 概要
{変更の目的・背景}

## 変更内容
- {具体的な変更点}

## 影響範囲
- {影響するファイル・機能}

## 動作確認・テスト結果
- [x] フロントエンドユニットテストがすべてパスする
- [x] バックエンドユニットテストがすべてパスする
- [x] ビルドが正常に完了すること

## 関連Issue
Fixes #{issue番号}

## チェックリスト
- [x] 自己レビューを行い、不要なデバッグコードやコメントが残っていないか確認した
- [x] 変数名や関数名がプロジェクトの命名規則に従っている
- [ ] ドキュメント（READMEやCLAUDE.md等）の更新が必要な場合、それらも修正した
EOF
)"
```

#### 既存のdraft PRを通常PRに変換する場合（並列実行時）

draft PR（テスト項目確認用）がすでに存在する場合は、以下の順で実行する:

1. `gh pr ready #{PR番号}` でdraftを解除する
2. **必ず** `gh pr edit #{PR番号}` でタイトルと本文を実装内容に合わせて更新する

```bash
gh pr ready #{PR番号}

# gh pr edit は Projects (classic) 廃止の影響でエラーになる場合がある。
# エラーが出た場合は必ず REST API で再実行すること（exit code 1 でも本文が更新されていないため）。
gh pr edit #{PR番号} \
  --title "{正式なタイトル（WIP: プレフィックスを外す）}" \
  --body "$(cat <<'EOF'
## 概要
{変更の目的・背景}

## 変更内容
- {具体的な変更点}

## 影響範囲
- {影響するファイル・機能}

## 動作確認・テスト結果
- [x] フロントエンドユニットテストがすべてパスする
- [x] バックエンドユニットテストがすべてパスする
- [x] ビルドが正常に完了すること

## 関連Issue
Fixes #{issue番号}

## チェックリスト
- [x] 自己レビューを行い、不要なデバッグコードやコメントが残っていないか確認した
- [x] 変数名や関数名がプロジェクトの命名規則に従っている
- [ ] ドキュメント（READMEやCLAUDE.md等）の更新が必要な場合、それらも修正した
EOF
)"
```

**`gh pr ready` だけで終わらせてはいけない。本文の更新は必須。**

`gh pr edit` がエラーになった場合は REST API で代替する（**エラー時は必ずこちらを実行すること**）:
```bash
gh api repos/{owner}/{repo}/pulls/#{PR番号} \
  --method PATCH \
  --field title="{タイトル}" \
  --field body="{本文（チェックリストのチェック状態も含めてすべて記入）}"
```
実行後 `--jq '.title'` 等でタイトルが正しく反映されたか確認すること。

---

### Step 7: 完了報告

以下のフォーマットで報告する:

```
## 作業完了

### 実施内容
- （変更した内容を箇条書きで列挙）

### テスト結果
- ビルド: 成功 / 失敗
- テスト: 成功（XX件） / 失敗（詳細）

### 既存テストの変更（該当する場合）
| ファイル | 変更箇所 | 変更理由 |
|---|---|---|
| path/to/test.ts | 変更内容の概要 | 仕様変更 / バグ修正 / テスト誤り |

### 次のアクション
- PR #{番号} のレビュー・マージをお願いします
```

---

## 注意事項

- **PR のマージはユーザーが実施する**。Claude はマージを実行しない
- `main` ブランチへの直接コミットは禁止
- スキーマ変更は必ず `db/schema.hcl` を編集してから `atlas schema apply` で適用する（`database.ts` の `initializeSchema` は使わない）
- 仕様が不明瞭な場合は推測せず、ユーザーに確認する
